# Export To CLO Contract

Статус: рабочий контракт для реализации
Источник: синтез по материалам из `F:\FURLAB\Info`

## Документы-основания

1. `FURLAB_edited_v17.docx`
- раздел `3.9 Интеграция системы FurLab с CLO 3D`
- раздел `3.8.4 Формирование спецификации изделия`
- раздел `3.8.5 Паспорт мехового материала`
- раздел `3.10.3 Экспорт отчёта`

2. `Приложение Z. Технический каркас плагина FurLab.docx`

## Назначение операции

`ExportToCLO` подготавливает результаты работы FurLab для дальнейшего использования в контуре `CLO 3D`.

Операция должна экспортировать:
- исходные контуры деталей;
- контуры фрагментов, полученных в результате выкладки;
- сведения о материалах, назначенных зонам/фрагментам;
- метаданные, связывающие детали, зоны, фрагменты и материалы.

На текущем этапе инвентарь в экспорт не входит.

## Состав экспортируемых сущностей

### 1. Исходные детали
Для каждой детали должны быть доступны:
- `detailId`
- `detailName`
- `detailContour`
- единицы измерения: `mm`

### 2. Зоны
Для каждой зоны должны быть доступны:
- `zoneId`
- `detailId`
- `zoneName`
- `zoneContour`
- `materialId` при назначении материала
- `napDirectionDeg`

### 3. Фрагменты выкладки
Для каждого фрагмента должны быть доступны:
- `fragmentId`
- `detailId`
- `zoneId`
- `fragmentContour`
- геометрические метрики фрагмента
- принадлежность к запуску/типу выкладки
- `materialId` через связанную зону

### 4. Материалы
Для каждого используемого материала должны быть доступны:
- `materialId`
- `materialName`
- `category`
- `species`
- полный набор свойств материала

Материал в экспорт включается как "паспорт мехового материала".

## Семантика данных

### Геометрия
- исходные детали экспортируются как исходные контуры;
- фрагменты экспортируются как результат клиппинга паттерна внутри `zoneContour`;
- экспорт должен сохранять реальную геометрию фрагментов, а не только параметры паттерна.

### Материал
- материал назначается на уровне зоны;
- фрагмент наследует материал своей зоны;
- в экспорт должен попадать полный набор свойств материала, а не только ссылка `materialId`.

### Направление
- экспорт должен сохранять направление, значимое для меховой сборки;
- для зоны должен быть доступен `napDirectionDeg`;
- если требуется, направление фрагмента вычисляется из зоны и режима выкладки.

## Минимальный экспортный пакет

Рабочая реализация должна собирать пакет, содержащий:
- геометрию исходных деталей;
- геометрию фрагментов;
- данные материалов;
- manifest, который связывает всё между собой.

Рекомендуемый логический состав:
- `original_parts`
- `layout_fragments`
- `materials`
- `manifest`

Точный файловый формат может быть выбран реализацией отдельно.

## Минимальный manifest

Минимальный manifest должен позволять восстановить связи:
- `detail -> zones`
- `zone -> material`
- `zone -> fragments`
- `fragment -> detail`

Минимальные поля manifest:
- `detailId`
- `zoneId`
- `fragmentId`
- `materialId`
- `layoutType`
- `paramsSnapshot` при необходимости

## Ограничения текущего этапа

На текущем этапе:
- инвентарь и scrap-логика в экспорт не включаются;
- экспорт ориентирован на детали, зоны, фрагменты и материалы;
- экспорт должен быть детерминированным для одинакового проекта и одинаковых входных параметров.

## Критерий готовности

Операция `ExportToCLO` считается реализованной на базовом уровне, если:
1. экспортируются все исходные детали проекта;
2. экспортируются все фрагменты активных выкладок;
3. каждый фрагмент связан с зоной и материалом;
4. материалы выгружаются с полным набором свойств;
5. пакет можно использовать как подготовленный контур передачи данных в `CLO 3D`.
