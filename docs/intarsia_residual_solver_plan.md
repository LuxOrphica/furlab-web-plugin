# Intarsia Residual Solver Plan

## Контекст

Текущая модель подбора для регулярной интарсии в режиме "из инвентаря" слишком грубая:

- подбор мыслится как `1 фрагмент -> 1 или 2 куска`;
- после первого куска у фрагмента могут оставаться несколько разорванных незакрытых областей;
- второй кусок может быть полезен только для части остатка, а не для всего фрагмента;
- при такой модели solver либо даёт плохое покрытие, либо пытается закрывать остатки наложением.

Вывод: решать нужно не задачу "подобрать 1-2 куска на фрагмент", а задачу
`фрагмент -> остаток покрытия -> последовательность кусков`.

## Цель

Переделать подбор для регулярной интарсии в итеративный residual-based solver:

- после каждого шага пересчитывать остаток покрытия;
- следующий кусок подбирать к остатку, а не к целому фрагменту;
- разрешить более 2 кусков на фрагмент, но останавливать solver по полезности, а не только по лимиту;
- исключить наложение кусков как способ "добрать" покрытие.

## Целевые критерии качества

Для oracle/live-case считаем режим рабочим, если:

- `intersections = 0`;
- `overlap = 0`;
- `matchedFragments = totalFragments`;
- `coverage >= 97-98%`;
- solver не использует "мусорные" добивки ради 1-2% площади;
- результат стабилен на повторных прогонах.

## Новая модель

### 1. Состояние фрагмента

Для каждого фрагмента хранить:

- `fragmentId`
- `fragmentMp`
- `coveredMp`
- `residualMp`
- `residualComponents`
- `selectedPieces`
- `coverageRatio`

### 2. Основной цикл

Вместо pair fallback:

1. выбрать лучший первый кусок;
2. обновить `coveredMp`;
3. посчитать `residualMp = fragmentMp - coveredMp`;
4. разбить остаток на компоненты;
5. выбрать следующую целевую компоненту остатка;
6. подобрать следующий кусок уже к этой компоненте;
7. повторять, пока есть полезный прирост.

### 3. Лимит кусков

Стартовый рабочий лимит:

- `maxPiecesPerFragment = 4`

Дальше можно расширять до 5 только по фактическим кейсам.

## Критерии остановки

Останавливать подбор для фрагмента, если выполнено одно из условий:

- достигнут `target coverage`;
- остаток меньше допустимого порога по площади;
- остаток состоит только из слишком узких/непрактичных зон;
- следующий кусок даёт слишком маленький прирост;
- превышен разумный лимит кусков;
- дальнейший шаг ухудшает качество остатка.

## Новый score

Оценка кандидата на каждом шаге должна включать:

- `coverageGain`
  - сколько новой площади остатка реально закрывает кусок;
- `outsidePenalty`
  - сколько выходит за границу целевого остатка/фрагмента;
- `overlapPenalty`
  - сколько конфликтует с уже выбранными кусками;
- `pieceCountPenalty`
  - штраф за добавление ещё одного куска;
- `residualComplexityPenalty`
  - штраф за "плохой" остаток после текущего шага.

## Residual complexity

Качество остатка после добавления куска оценивать как минимум по:

- числу компонент остатка;
- площади крупнейшей компоненты;
- числу крупных компонент;
- минимальной толщине компонент;
- числу узких хвостов и карманов.

Плохой первый кусок может давать высокий текущий coverage, но оставлять плохой residual. Такие варианты должны проигрывать.

## Этапы внедрения

### Этап 1. Residual loop без сложной эвристики

Сделать рабочий итеративный solver:

- residual после каждого шага;
- подбор следующего куска по `coverageGain`;
- до 4 кусков на фрагмент;
- без наложений.

Результат этапа:

- корректный цикл подбора;
- корректная геометрия остатка;
- базовая диагностика по шагам.

### Этап 2. Качество остатка

Добавить в score:

- штраф за большое число компонент остатка;
- штраф за крупный незакрытый остаток;
- штраф за узкие остатки;
- штраф за бесполезные дополнительные куски.

Результат этапа:

- solver начинает выбирать не просто "лучший текущий кусок", а лучший ход с точки зрения всего фрагмента.

### Этап 3. Диагностика

Для каждого `fragmentId` сохранять пошаговую диагностику:

- какой кусок выбран;
- на каком шаге;
- какой был прирост покрытия;
- какой остаток остался после шага;
- почему отклонены сильные альтернативы;
- сколько компонент остатка было до и после.

Результат этапа:

- можно объяснить любое плохое решение solver без гадания по скриншотам.

### Этап 4. Автотесты

Добавить oracle/live regression tests:

- кейс должен воспроизводиться автоматически;
- тест проверяет:
  - `matchedFragments`
  - `coverage`
  - `overlap`
  - `intersections`
- отдельный тест на residual-подбор при `3+` кусках на фрагмент.

## Минимальный первый инкремент

Первый рабочий инкремент должен сделать только это:

1. residual loop;
2. до 4 кусков на фрагмент;
3. пересчёт остатка после каждого шага;
4. запрет наложений;
5. пошаговый debug output.

Без попытки сразу "идеально" настраивать веса.

## Главное правило

Solver должен оптимизировать не просто текущее покрытие куском, а траекторию уменьшения остатка.

Коротко:

- не `best piece for fragment`;
- а `best next move for residual`.
