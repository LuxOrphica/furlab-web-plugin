# FurLab — Солвер автоматической укладки

Документация алгоритмов решения задачи оптимального раскроя меха.

---

## 1. Постановка задачи

### Формально

Дано:
- Зона Z — произвольный полигон площадью S(Z)
- Инвентарь P = {p₁, p₂, …, pₙ} — набор кусков меха, каждый с контуром и направлением ворса
- Допуск ворса φ — максимальное отклонение направления ворса детали от целевого

Требуется найти раскладку R = {(pᵢ, xᵢ, yᵢ, θᵢ)} такую, что:

```
Покрытие:   S(Z \ ⋃ pᵢ) → min    (непокрытая площадь → 0)
Отходы:     S(⋂ пересечений) → min (перекрытия → 0)
Экономия:   |R| → min              (количество деталей минимально)
Ворс:       |napᵢ + θᵢ − φ_target| ≤ φ  для всех i
```

### Почему задача сложная

Set Cover (покрыть зону деталями) — **NP-hard** в общем случае.
Точного полиномиального алгоритма не существует.
Практическое решение — эвристика: не гарантирует оптимум, но даёт хорошее решение за приемлемое время.

Ограничение ворса существенно сужает пространство поиска:
при допуске φ = 15° доступный диапазон поворотов уменьшается в 12 раз по сравнению
с задачей без ограничений.

## 2. Реализованный алгоритм: жадный с растровым покрытием (`cover_grid_solver.js`)

### Идея

Зона растеризуется в сетку ячеек размером `r мм`.
Алгоритм итеративно выбирает деталь и позицию,
максимизирующие число новых покрытых ячеек.

### Структуры данных

```
zoneMask      — битсет ячеек внутри зоны
coveredMask   — битсет уже покрытых ячеек
uncoveredMask — zoneMask \ coveredMask
```

Битсеты хранятся как `Uint32Array` — 32 ячейки на одно слово.
Операции AND, OR, XOR выполняются побитово над словами — эффективно для больших сеток.

### Алгоритм

```
for iter = 1..maxPieces:
  target ← firstUncoveredCell(uncoveredMask)   // якорная точка
  best ← null

  for each template tpl in candidates:
    if tpl already used: skip
    for angle a in [napTarget − tol .. napTarget + tol], step angleStep:
      rotated ← rotate(tpl.centered, a)
      contour ← translate(rotated, target)
      win ← rasterize(contour, gridSpec)
      s ← evaluate(win, zoneMask, coveredMask)
      score ← gainCount − penalties(overlap, outside, nap)
      if score > best.score: best ← (tpl, contour, a, score)

  if best found:
    commitWindow(best.win, coveredMask, uncoveredMask)
    used.add(best.tpl.key)
    placements.append(best)
  else:
    noProgressStreak++
    if noProgressStreak > threshold: break
```

### Функция оценки позиции

```
gainCount    = |win ∩ coverageMask ∩ ~coveredMask|   // новые покрытые ячейки ядра
insideCount  = |win ∩ zoneMask|                       // ячейки внутри зоны
overlapCount = |win ∩ zoneMask ∩ coveredMask|         // перекрытие с уже уложенными
outsideCount = |win \ zoneMask|                       // ячейки вне зоны

score = gainCount
      − overlapPenalty · overlapCount
      − outsidePenalty · outsideCount
      − napPenalty · (dNap / napTol)
      − utilizationPenalty · (1 − gainCount/insideCount)
```

### Параметры

| Параметр | Смысл | Значение по умолчанию |
|---|---|---|
| `rasterMm` | размер ячейки сетки | 2 мм |
| `maxPieces` | максимум итераций | 240 |
| `hardMaxSolveMs` | таймаут | 600 000 мс |
| `coverageTarget` | целевое покрытие | 99.99% |
| `overlapPenalty` | штраф за перекрытие | 0.25 |
| `outsidePenalty` | штраф за выход за зону | 0.05 |
| `napWeight` | вес штрафа ворса | 1.0 |
| `angleStep` | шаг перебора углов | 3–12° |

### Известные ограничения

| Проблема | Причина | Следствие |
|---|---|---|
| Каждая деталь используется ≤ 1 раза | `used.add(key)` после первого размещения | Повторные экземпляры в инвентаре игнорируются |
| Якорь — первая непокрытая ячейка | Детерминированный выбор без глобального анализа | Поздние итерации получают неудобные остатки |
| Нет отката | Жадность без backtracking | Застревает в локальном оптимуме |
| Грубый шаг угла | 3–12° вместо непрерывного поиска | Упускает точные угловые позиции |

### Сложность

```
O(maxPieces · |candidates| · (2·napTol/angleStep) · rasterCost)
```

`rasterCost` — стоимость растеризации одного контура на сетку,
пропорциональна числу ячеек в bounding box детали.

## 3. Анализ: почему жадный проигрывает ручной укладке

Ручная укладка опытным мастером превосходит жадный солвер
по трём метрикам: количество деталей, процент покрытия, процент отходов.

### Корень проблемы

Жадный алгоритм решает задачу **Coverage** («закрой все ячейки»),
тогда как реальная задача — **Irregular Nesting** («уложи минимум деталей с минимальными отходами»).
Это принципиально разные постановки.

### Механизм деградации

```
Итерация 1–5:   крупные детали ложатся хорошо
Итерация 6–15:  оставшиеся области — неудобные L-образные остатки
Итерация 16+:   ни одна деталь из инвентаря не влезает в остаток
                → noProgressStreak достигает порога → выход
                → зона покрыта на 40–60%
```

Мастер видит всю зону сразу и планирует последовательность заранее.
Жадный алгоритм видит только текущую непокрытую точку.

## 4. Предлагаемый подход: метод имитации отжига (Simulated Annealing)

### Принцип

Вместо пошагового добавления — **глобальная оптимизация состояния**.

```
Состояние = полная раскладка [(деталь, x, y, θ), ...]
Энергия   = мера «плохости» раскладки (минимизируется)
```

Алгоритм допускает временные ухудшения — это позволяет выбираться
из локальных минимумов, недостижимых для жадного метода.

### Функция энергии

```
E(R) = w₁ · S_непокрыто(R)    // w₁ = 1000  — главный приоритет
     + w₂ · S_перекрытий(R)   // w₂ = 10    — перекрытия деталей
     + w₃ · |R|                // w₃ = 1     — количество деталей

где S_непокрыто = S(Z \ ⋃ᵢ pᵢ)   — площадь незакрытой зоны
    S_перекрытий = Σᵢⱼ S(pᵢ ∩ pⱼ) — суммарные перекрытия
```

Соотношение весов кодирует приоритеты:
сначала закрыть всю зону, потом убрать перекрытия, потом сократить детали.

### Алгоритм

```
R₀ ← warm_start(жадный_солвер)    // начинаем не с нуля
T  ← T₀                            // начальная "температура"

while T > T_min and time < budget:

  ход ← случайный_ход(R)          // выбрать тип возмущения
  R'  ← применить(ход, R)

  ΔE ← E(R') − E(R)

  if ΔE < 0:                       // улучшение → принять
    R ← R'
  else:                            // ухудшение → принять с вероятностью
    if random() < exp(−ΔE / T):
      R ← R'

  T ← T · α                       // остывание (α ≈ 0.995)

return R
```

**Критерий Метрополиса** `exp(−ΔE / T)`:
- При высокой T (горячо) — принимает почти любые ходы, исследует широко
- При низкой T (холодно) — принимает только улучшения, сходится к локальному оптимуму

### Типы ходов

| Ход | Описание | Вероятность выбора |
|---|---|---|
| Сдвиг | переместить деталь на (dx, dy) | 35% |
| Поворот | повернуть деталь на ±Δθ в пределах допуска ворса | 25% |
| Замена | заменить деталь на другую из инвентаря | 20% |
| Удаление | убрать деталь если зона остаётся покрытой | 10% |
| Добавление | поместить новую деталь в непокрытую область | 10% |

### Расписание охлаждения

```
T₀   = 0.1 · E(R₀)        // начальная температура = 10% от начальной энергии
T_min = 0.001 · T₀         // порог остановки
α    = 0.995               // коэффициент охлаждения
```

При α = 0.995 температура падает в 2 раза каждые ~139 шагов.
Типичное число шагов: 10 000–100 000 в зависимости от размера зоны.

### Учёт ворса

```
Допустимое множество углов для детали i:
  Θᵢ = { θ : |nap_i + θ − φ_target| ≤ φ_tol }

Ход "поворот" выбирает θ ∈ Θᵢ равномерно.
Ход "замена" проверяет совместимость ворса новой детали.
```

Ограничение ворса сужает пространство поиска — алгоритм сходится быстрее.

### Warm Start

Жадный солвер используется для построения начального решения R₀.
Отжиг улучшает R₀, а не строит раскладку с нуля.

```
E(R₀) << E(случайной раскладки)
→ алгоритм начинает ближе к оптимуму
→ требуется меньше итераций для сходимости
```

### Ожидаемое качество

| Метод | Покрытие | Отходы | Деталей |
|---|---|---|---|
| Жадный (текущий) | 40–60% | высокие | мало |
| Жадный + warm start | 60–75% | средние | — |
| **Отжиг (предлагается)** | **85–95%** | **низкие** | **меньше** |
| Теоретический оптимум | 100% | 0 | минимум |

## 5. Сравнение с альтернативными подходами

### Генетический алгоритм

```
Популяция = N раскладок (особей)
Скрещивание: взять часть позиций от R₁, часть от R₂
Мутация: случайное возмущение одной детали
Отбор: выжившие = top-K по fitness = 1/E
```

**Преимущества:** хорошо исследует пространство, параллелизуется.

**Недостатки для данной задачи:**
- N полных раскладок в памяти → N-кратный расход ресурсов
- Скрещивание раскладок нетривиально (детали пересекаются, нарушается ворс)
- Полигональные операции дорогие → N=20 особей = ×20 к стоимости итерации

### NFP (No-Fit Polygon)

Стандарт промышленных CAD-систем раскроя.

```
NFP(A, B) = граница всех позиций B относительно A,
            при которых A и B не пересекаются
```

Позволяет точно вычислить допустимые позиции для каждой пары деталей.

**Преимущества:** математически точен, даёт плотную упаковку.

**Недостатки:**
- Вычисление NFP для произвольных полигонов — сложная геометрическая операция
- Для N деталей нужно O(N²) NFP → при N=100 это 10 000 операций
- Реализация non-trivial, особенно для вогнутых контуров

### Вывод

Для данной задачи оптимален **отжиг с warm start**:
- Не требует популяции → экономия памяти
- Работает с одной раскладкой → можно позволить дорогие полигональные операции
- Ограничение ворса уменьшает пространство поиска → ускоряет сходимость
- Хорошо изучен теоретически, предсказуемое поведение

## 6. NP-сложность и практические следствия

Set Cover (частный случай задачи покрытия) — NP-hard.
Это означает:

1. **Нет полиномиального алгоритма** нахождения точного оптимума (если P ≠ NP)
2. **Гарантия приближения**: жадный алгоритм для Set Cover даёт решение
   не хуже чем `(1 + ln k) · OPT`, где k — максимальный размер подмножества
3. **Практический вывод**: эвристика — это не компромисс, это единственный
   разумный подход для задач реального размера

Ограничение ворса (φ ≤ 15°) превращает непрерывную задачу в
существенно дискретную — это улучшает практическую сходимость эвристик.

## 7. Направления развития: нарезка кусков (Cutting Stock)

### Идея

Куски меха в инвентаре **делимые** — их можно разрезать на части
при условии соблюдения минимального размера каждой части.
Остатки после нарезки становятся новыми кусками и пополняют инвентарь.

Это потенциально резко улучшает покрытие: вместо того чтобы искать кусок
подходящей формы — взять большой кусок и отрезать нужное.

### Почему это отдельная задача

```
Текущая задача:   уложить готовые куски → Irregular Nesting (NP-hard)
С нарезкой:       решить как резать + куда класть → Cutting Stock + Nesting
```

Это **двухуровневая** задача:

```
Уровень 1 (внешний): какие куски резать и по каким линиям?
Уровень 2 (внутренний): как уложить полученные части в зону?
```

Каждый уровень сам по себе NP-hard. Совместная оптимизация —
принципиально сложнее каждого из них по отдельности.

### Ограничения специфики меха

- **Минимальный размер** части после разреза — задаётся параметром `fragmentMinAlongMm` / `fragmentMinAcrossMm`
- **Форма разреза** — следует контуру фрагмента зоны, не обязательно прямая линия
- **Направление разреза** — без жёстких ограничений (припуски перекрывают шов)

### Почему ранняя реализация не работала

Попытка решить обе задачи одним солвером приводит к тому,
что алгоритм не имеет чёткой структуры ни для одной из них.
Задача теряет форму и становится неуправляемой.

**Правильный подход** — декомпозиция:

```
Шаг 1: запустить нестинг на целых кусках
Шаг 2: найти непокрытые области (остатки зоны)
Шаг 3: для каждого остатка найти кусок который можно разрезать
        так чтобы одна часть покрыла остаток, другая ≥ minSize
Шаг 4: добавить нарезанные части в инвентарь и повторить нестинг
```

Это итеративный процесс, каждый шаг которого — отдельно решаемая задача.
