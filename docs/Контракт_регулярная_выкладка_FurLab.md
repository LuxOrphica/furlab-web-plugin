# КОНТРАКТ / ТЕХНИЧЕСКОЕ ЗАДАНИЕ
## на разработку алгоритма регулярной выкладки с подбором кандидатов для FurLab

**Назначение документа.** Настоящий документ фиксирует требования к алгоритму регулярной выкладки с подбором кандидатов из инвентаря для последующего согласования и утверждения.

## 1. Предмет работ

Исполнитель разрабатывает и передает Заказчику алгоритм подбора кандидатов для режима регулярной выкладки в системе FurLab, а также логику ранжирования, фильтрации и выбора кусков материала из инвентаря для заполнения регулярных фрагментов зоны.

Результат работ должен обеспечивать автоматизированный подбор пригодных кусков материала (ScrapPiece) для регулярных фрагментов, сформированных внутри целевой зоны (Zone), с учетом геометрии фрагментов, ограничений по материалу, направлению ворса, допустимому повороту, статуса куска и качества.

## 2. Цель разработки

Целью работ является повышение качества автоматической выкладки в режиме регулярной схемы за счет перехода от примитивного подбора по площади или габаритам к алгоритму, который:
- классифицирует регулярные фрагменты по типу;
- отбирает кандидатов с учетом геометрической пригодности;
- учитывает ограничения по ворсу и запрет зеркального отражения;
- минимизирует отход, скрытую площадь и неэффективное использование ценных кусков;
- обеспечивает объяснимый результат подбора и возможность предпросмотра.

## 3. Термины и сущности

В рамках настоящего ТЗ используются следующие сущности:
- **Zone** — целевая зона размещения;
- **Fragment** — регулярный фрагмент, полученный после построения сетки или паттерна и клиппинга по зоне;
- **ScrapPiece** — кусок материала из инвентаря;
- **RegularLayout** — режим регулярной выкладки;
- **LayoutRun** — запуск алгоритма выкладки;
- **LayoutRunScrapPlacement** — результат назначения куска на фрагмент;
- **napDirectionDeg** — направление ворса куска;
- **napToleranceDeg** — допустимое отклонение по направлению ворса;
- **insideRatio** — доля площади куска, приходящаяся на рассматриваемый фрагмент после размещения;
- **coverageRatio** — доля площади фрагмента, покрываемая кандидатом;
- **wasteRatio** — доля площади, уходящая в отход или обрезку;
- **flip / mirror** — зеркальное отражение, запрещенное для физического куска;
- **rotation** — допустимый поворот.

## 4. Общие требования к алгоритму

### 4.1. Входные данные
Алгоритм должен принимать на вход:
- контур зоны (zoneContour);
- параметры регулярной сетки или шаблона фрагментации;
- список доступных ScrapPiece из инвентаря;
- ограничения по материалу, качеству, статусу и направлению ворса;
- настройки подбора из конфигурации режима RegularLayout.

### 4.2. Выходные данные
Алгоритм должен формировать:
- список кандидатов для каждого фрагмента;
- итоговое назначение выбранных кусков на фрагменты;
- служебные метрики по каждому назначению;
- данные для предпросмотра;
- данные для сохранения в LayoutRunScrapPlacement.

## 5. Функциональные требования

### 5.1. Формирование фрагментов
На первом этапе алгоритм должен:
1. построить регулярную сетку или паттерн внутри зоны;
2. выполнить клиппинг по реальному контуру зоны;
3. исключить или нормализовать слишком мелкие и непригодные фрагменты согласно параметрам минимальной ширины, минимальной длины, минимальной площади, правилам объединения мелких фрагментов и допустимому упрощению контура.

### 5.2. Расчет признаков фрагмента
Для каждого фрагмента должны быть рассчитаны как минимум: площадь, bounding box, ширина и высота, aspect ratio, rectangularity, convexity, доля контакта с границей зоны, сложность контура и положение внутри зоны.

### 5.3. Классификация фрагментов
Каждый фрагмент должен быть автоматически отнесен к одному из классов: внутренний, краевой, угловой, низкоприоритетный (в том числе мелкий). Классификация должна использовать геометрические признаки фрагмента и степень его подрезки границей зоны.

### 5.4. Предфильтрация кандидатов
Перед точным сравнением алгоритм должен отбрасывать неподходящие ScrapPiece по жестким условиям:
- материал не совпадает;
- статус не Available;
- качество ниже допустимого;
- направление ворса не укладывается в napToleranceDeg;
- требуется зеркальное отражение;
- габариты или площадь явно недостаточны.

### 5.5. Геометрическая оценка кандидатов
Для каждого кандидата по отношению к каждому фрагменту должны рассчитываться: coverageRatio, insideRatio, wasteRatio, rotationDeg, napDeltaDeg, shapeMismatch и hiddenOverlapRisk либо иной показатель неэффективного использования.

### 5.6. Ранжирование кандидатов
Для каждого фрагмента должен формироваться ранжированный список кандидатов на основе интегрального score:

**Score =** `w1 × coverageRatio + w2 × insideRatio − w3 × wasteRatio − w4 × hiddenOverlapRisk − w5 × shapeMismatch − w6 × rotationPenalty − w7 × scarcityPenalty + w8 × classAffinity`

Конкретные значения весов `w1–w8` определяются конфигурацией алгоритма и фиксируются в версии настроек.

### 5.7. Логика выбора
Алгоритм должен выполнять назначение кандидатов не в случайном порядке, а по приоритету сложности фрагментов. Сначала должны обрабатываться сложные угловые фрагменты, затем краевые фрагменты, затем крупные внутренние фрагменты, после чего мелкие низкоприоритетные фрагменты. Для первой версии допускается жадная стратегия при условии соблюдения указанного порядка приоритетов.

## 6. Специальные ограничения

### 6.1. Ограничения по ворсу
Алгоритм обязан учитывать направление ворса. Должна использоваться единая конвенция углов: `0° от +X`, `clockwise` при `Y вниз`. Размещение допускается только при соблюдении `napToleranceDeg`.

### 6.2. Запрет зеркального отражения
Алгоритм не должен использовать flip или mirror для ScrapPiece. Допустимы только повороты.

### 6.3. Бережное использование сложных кусков
Сложные по форме куски не должны в приоритетном порядке расходоваться на простые внутренние прямоугольные фрагменты, если имеются более подходящие простые кандидаты.

### 6.4. Объяснимость результата
Для каждого назначения должны быть доступны причины выбора: лучшее покрытие, меньше отход, лучшее совпадение формы, соблюдение ворса, штраф за редкость куска и иные факторы, влияющие на итоговый score.

## 7. Требования к результату разработки

Исполнитель должен передать Заказчику:
1. описание алгоритма в текстовом виде;
2. перечень входных параметров и их смысл;
3. формулу ранжирования;
4. правила классификации фрагментов;
5. псевдокод либо код реализации;
6. формат выходных данных;
7. примеры сценариев работы;
8. перечень ограничений и допущений.

Если работы включают программную реализацию, дополнительно передаются исходный код, комментарии к ключевым модулям, описание точек интеграции и инструкция по запуску и тестированию.

## 8. Этапы выполнения работ

**Этап 1. Аналитика и проектирование**
- анализ существующей логики регулярной выкладки;
- описание целевого пайплайна;
- определение признаков фрагментов и кусков;
- описание score-модели.

Результат этапа: проект алгоритма и согласованная спецификация.

**Этап 2. Реализация базовой версии**
- генерация признаков;
- классификация фрагментов;
- предфильтрация кандидатов;
- локальный скоринг;
- жадное назначение по приоритету сложности.

Результат этапа: рабочая MVP-версия алгоритма.

**Этап 3. Отладка и тестирование**
- проверка на типовых зонах;
- сравнение качества с текущей логикой;
- корректировка весов и порогов;
- подготовка примеров.

Результат этапа: стабилизированная версия и отчет по тестам.

**Этап 4. Опциональные улучшения**
- beam search;
- weighted matching;
- расширенная explainability;
- ручная корректировка кандидатов в интерфейсе.

Результат этапа: улучшенная версия, если предусмотрена договором.

## 9. Критерии приемки

Работа считается принятой, если одновременно выполняются следующие условия:
1. алгоритм формирует список кандидатов для регулярных фрагментов;
2. отбор учитывает материал, статус, качество и ворс;
3. зеркальное отражение не используется;
4. внутренние простые фрагменты преимущественно получают простые по форме куски;
5. краевые и угловые фрагменты получают более релевантных кандидатов, чем при подборе только по площади;
6. по каждому назначению доступны ключевые метрики и причина выбора;
7. результат воспроизводим при одинаковых входных данных;
8. переданы все материалы, указанные в разделе 7.

## 10. Особое правило припуска

Для режима регулярной выкладки припуск применяется к куску материала, а не ко всей зоне.

Это означает:
- зона не должна уменьшаться или модифицироваться из-за припуска;
- регулярные фрагменты не должны уменьшаться или модифицироваться из-за припуска;
- для расчета полезного покрытия должна использоваться внутренняя рабочая область самого куска, сформированная через inward offset контура куска на величину `pieceSeamReserveMm`;
- визуально при предпросмотре может отображаться полный контур куска, однако в расчет `gain`, `coverageRatio` и иных метрик покрытия должна входить только рабочая область куска с учетом припуска.

Данное правило является обязательным и должно применяться во всех сценариях расчета, подбора и оценки результата регулярной выкладки.

## 11. Приложение A. Интеграционные параметры приемки (по согласованию сторон)

Раздел 11 является приложением к алгоритмическому ТЗ и применяется при внедрении (интеграция, эксплуатационная приемка, KPI). Базовые требования к алгоритму определяются разделами 1–10.

### 11.1. Количественные KPI (целевые)
Для каждого тестового набора должны фиксироваться и сравниваться следующие показатели:
- `coveragePct` — процент покрытия регулярных фрагментов;
- `matchedFragmentsPct` — доля фрагментов с назначенным кандидатом;
- `wastePct` — доля отхода;
- `meanFitScore` и `p10FitScore` — средний и 10-й перцентиль качества назначения;
- `runtimeMs` (P50 / P95) — время расчета.

Целевые ориентиры для MVP (подлежат подтверждению протоколом испытаний):
- `coveragePct` ориентировочно не ниже `92%` на согласованном наборе кейсов;
- `matchedFragmentsPct` ориентировочно не ниже `90%`;
- `runtimeMs(P95)` ориентировочно в пределах `15000` мс при `N_fragments <= 120`, `N_candidates <= 300`;
- отсутствие нарушений ограничений по ворсу и зеркалу (`mirror/flip = false`).

Примечание: окончательные пороги фиксируются только после согласования тестового датасета, baseline-версии и условий запуска.

### 11.2. Обязательный fallback-контракт
Если базовый подбор не дал назначения для части фрагментов, система обязана:
1. вернуть диагностический код причины (`insufficient_candidates_for_fragments`, `nap_violation`, `size_gate`, и т.п.);
2. выполнить переход к резервной стратегии подбора, определенной конфигурацией запуска и протоколом согласования;
3. вернуть итоговую стратегию в поле `strategyUsed`;
4. не скрывать частичный результат: все неназначенные фрагменты должны иметь статус `needs_attention` с причиной.

### 11.3. Воспроизводимость результата
Для каждого `LayoutRun` обязательно сохранять:
- `algorithmVersion`;
- `weightsVersion`;
- `randomSeed` (если используется стохастика);
- `buildId` сервера;
- входные параметры запуска.

Результат считается воспроизводимым, если при одинаковом входе и одинаковых версиях артефактов формируется идентичный набор назначений.

### 11.4. Формальный JSON-контракт
Исполнитель обязан приложить к реализации:
- JSON-схему входного запроса подбора;
- JSON-схему результата (`placements`, `kpi`, `diagnostics`, `strategyUsed`);
- справочник диагностических кодов и их трактовку для интерфейса.

Это требование обязательно для исключения расхождений между сервером и UI.

### 11.5. Базовый набор приемочных тестов
До приемки должны быть выполнены и приложены:
- не менее 10 типовых кейсов по разным зонам и размерам сетки;
- не менее 3 стресс-кейсов (повышенная сложность контура/ограниченный инвентарь);
- сравнение с текущей логикой по каждому кейсу (таблица «было/стало»).

Минимум по отчету:
- входные параметры;
- итоговые KPI;
- список неназначенных фрагментов и причин;
- время выполнения.

## 12. Приложение B. Требования к интеграции API и телеметрии

Раздел 12 относится к серверному API и UI-каналу выполнения и не изменяет алгоритмические требования разделов 1–10.

### 12.1. Телеметрия прогресса
В процессе расчета должны публиковаться этапы и частичные метрики:
- текущая фаза (`phase`);
- количество обработанных фрагментов/кандидатов;
- промежуточные `coveragePct`, `matchedFragmentsPct`, `runtimeMs`;
- признаки стагнации (`no_progress`, `time_budget_exceeded`).

Порог отсутствия новых событий определяется конфигурацией исполнения; для MVP рекомендуется значение 3 секунды. При достижении порога должен возвращаться явный статус канала телеметрии (`stale`/`waiting`) без потери финального результата.

### 12.2. Совместимость UI/сервер
UI обязан корректно обрабатывать частичный и финальный результат, включая:
- `strategyUsed`;
- `diagnostics` и справочник кодов;
- `placements` со статусами `matched`/`needs_attention`;
- `kpi` для итогового экрана приемки.
